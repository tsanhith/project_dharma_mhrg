# Project Dharma MHRG - Deep Architecture Documentation (Phase 4)

This log outlines the architectural decisions and implementation of Phase 4: ""Robot 2"" (The Surgeon). It details how we use free models to intelligently slice raw text and inject it into a structured layout with zero human intervention.

## Phase 4: Segment B - The Surgeon (Resume Tailor)

### The Goal
Take the massive, unstructured `raw_jd_text` generated by the Scout (Robot 1) and condense it into a perfect, ATS-compliant JSON structure. This JSON will later be injected directly into a LaTeX document template (via Jinja2) and compiled to PDF. It must do this strictly using the 100% Free architecture rules.

### Implementation Details & File Architecture

*   **gents/tailor.py**:
    *   **The Message Map**: The file declares @celery_app.task(name='agents.tailor.tailor_resume'). This physically binds it to the 	ailor_queue in Redis. When the Scout finishes scraping, it blindly throws a job_id into this queue, allowing the Surgeon to pick it up immediately.
    *   **Idempotency & Pre-flight Checks**: The very first operation is fetching the job from Postgres and verifying job.status == JobStatus.SCRAPED. If the web-scraper failed and left it in PENDING or ERROR, the Surgeon gracefully exits and refuses to run expensive LLM processes on bad data.
    
    *   **The LLM Engine (Groq Over OpenAI)**:
        *   To comply with the Zero-Cost mandate, we invoke the groq SDK instead of openai. 
        *   We specifically target the llama3-8b-8192 model. It possesses lightning-fast inference (hundreds of tokens per second) and is perfectly capable of resume extraction.
    
    *   **Context Window Optimization (String Slicing)**:
        *   Websites are bloated with copyright footers, nav-bar text, and tracking jargon. We enforce job.raw_jd_text[:3000]. By aggressively slicing the text to the first 3000 characters, we prevent Groq token-limit explosions while capturing the vast majority of relevant job requirements.
        
    *   **Forcing Structured Output (esponse_format)**:
        *   LLMs are notoriously chatty ("Here are your resume changes!"). We enforce "type": "json_object" inside the API call to physically prevent the LLM from outputting conversational text. 
        
    *   **In-Memory Gatekeeping (Pydantic validation)**:
        *   We declared class BulletPoints(pydantic.BaseModel) directly in the agent.
        *   Once the LLM returns the string response, we immediately pass it through BulletPoints.model_validate_json(). If the LLM hallcuinated the JSON key (e.g. returning {"bullet_pts": ...} instead of {"bullets": ...}), Python throws a ValidationError instantly, avoiding a fatal compilation error deep inside the LaTeX engine.

    *   **The Hand-Off**:
        *   Assuming successful parsing, it bumps the Postgres State Machine to JobStatus.TAILORING. 
        *   Finally, it fires the ticket to sync_queue so Robot 3 (The Command Center) can intercept it.
        
    *   **Error Trapping**:
        *   Using deep 	ry/except chains, both API connectivity issues and pydantic.ValidationError hits are caught. 
        *   It will write the exact failure reason (e.g. "LLM returned invalid JSON") to the error_message DB column and safely kill the transaction, never crashing the parent Celery thread.
